# バックエンドアーキテクチャの時系列まとめ

このドキュメントは、
**MVC → 三層 → クリーン → マイクロサービス / イベント駆動**
という流れを「なぜ生まれたか」「何を解決したか」の軸で整理する。

---

## 1. 全体の時系列マップ

```
1970s    1990s    2000s        2010s〜現在
 MVC  →  三層  →  レイヤード  →  クリーン
                               ↓
                    マイクロサービス / イベント駆動
```

※ 新しいものほど「スケール」「変更耐性」「分散」を意識

---

## 2. MVCアーキテクチャ（最初の整理）

### 登場背景（1970s）

- 画面・ロジック・データが混在していた
- UIを整理したい

### 構造

```
Model / View / Controller
```

### 主眼

- **画面（UI）の整理**

### 特徴

- View（HTML）が存在
- サーバーサイドレンダリング前提

### 現代での立ち位置

- Web画面（Thymeleaf / JSP）では現役
- REST APIでは主流ではない

---

## 3. 三層アーキテクチャ（業務システム時代）

### 登場背景（1990s）

- 業務システムが巨大化
- DB直結コードの破綻

### 構造

```
Presentation
Business
Data Access
```

### 主眼

- **責務分離**

### 特徴

- UI・業務・DBを分離
- 業務ロジックが中心

### 現代での立ち位置

- REST APIの王道

---

## 4. レイヤードアーキテクチャ（三層の実装形）

### 目的（2000s）

- 三層を実装しやすく

### Spring Boot例

```
Controller
Service
Repository
DAO
```

### 主眼

- **実装整理**

---

## 5. クリーンアーキテクチャ（変更耐性）

### 登場背景（2010s）

- 変更に弱い
- フレームワーク依存
- テスト困難

### 思想構造

```
Entities
UseCases
Interface Adapters
Frameworks
```

### 主眼

- **依存方向の制御**

### 特徴

- 内側は外側を知らない
- ビジネスルール最優先

### 現代での立ち位置

- 中〜大規模向け
- 小規模では過剰

---

## 6. マイクロサービスアーキテクチャ（分割戦略）

### 登場背景（2010s）

- モノリスの限界
- 組織・チーム拡大

### 構造

```
Service A
Service B
Service C
```

### 主眼

- **システム分割・独立性**

### 特徴

- サービス単位で独立
- DBも原則分離

### 注意

- 小規模では過剰

---

## 7. イベント駆動アーキテクチャ（疎結合）

### 登場背景

- 同期呼び出しの限界
- 耐障害性不足

### 構造

```
Producer → Event → Broker → Consumer
```

### 主眼

- **疎結合・非同期**

### 特徴

- 最終的整合性
- 拡張しやすい

---

## 8. マイクロサービス × イベント駆動

### なぜ組み合わさる？

- サービス間同期は危険
- イベントでつなぐと安全

### 実務構成

```
[Service]
  └ 内部：三層 or クリーン
```

---

## 9. 全体比較まとめ

| 時代    | アーキテクチャ   | 主な目的 |
| ------- | ---------------- | -------- |
| 1970s   | MVC              | UI整理   |
| 1990s   | 三層             | 責務分離 |
| 2000s   | レイヤード       | 実装整理 |
| 2010s   | クリーン         | 変更耐性 |
| 2010s〜 | マイクロサービス | 分割     |
| 2010s〜 | イベント駆動     | 疎結合   |

---

## 10. 一言で覚える

- MVC：画面のため
- 三層：業務のため
- クリーン：未来のため
- マイクロサービス：分割のため
- イベント駆動：疎結合のため

---

この時系列を理解すれば、
アーキテクチャ名に振り回されなくなる。
