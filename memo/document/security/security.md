# Webシステムに関する基本的なセキュリティ用語まとめ

## 1. CORS (Cross-Origin Resource Sharing)

- **意味**: ブラウザが「別ドメインのサーバーに勝手にアクセスするのを制限する仕組み」
- **例**:
  - フロント: `https://myapp.com`
  - 外部API: `https://external.com`
  - この場合、ブラウザが直接呼ぶと制約でブロックされる
- **対策**: サーバー間通信にする（バックエンド経由）

---

## 2. XSS (Cross-Site Scripting)

- **意味**: 攻撃者が悪意のあるスクリプトをウェブサイトに埋め込み、他のユーザーのブラウザで実行させる攻撃
- **被害例**:
  - クッキー盗まれる → ログイン情報流出
  - 画面が改ざんされる
- **対策**:
  - ユーザー入力をそのままHTMLに表示しない
  - HTMLエスケープやライブラリの安全なレンダリングを利用

---

## 3. CSRF (Cross-Site Request Forgery)

- **意味**: ユーザーが意図しないリクエストを別サイトから送らされる攻撃
- **被害例**:
  - ログイン中に他サイトから送られたリクエストで勝手にデータが変更される
- **対策**:
  - リクエストに CSRFトークン を含める
  - SameSite属性をクッキーに付与する

---

## 4. SQLインジェクション

- **意味**: 攻撃者がSQL文に悪意あるコードを挿入し、DBを不正操作する攻撃
- **被害例**:
  - データ閲覧、削除、改ざん
- **対策**:
  - プレースホルダーやORMを使ってSQLを組み立てる
  - ユーザー入力を直接SQLに埋め込まない

---

## 5. APIキーや認証情報の漏洩

- **意味**: 外部APIにアクセスする秘密情報がブラウザや公開場所に置かれている状態
- **被害例**:
  - 誰でもAPIを呼び出せる → 外部サービスの不正利用
- **対策**:
  - バックエンドで管理し、フロントには渡さない
  - 環境変数やシークレット管理サービスを利用

---

## 6. HTTPS / SSL

- **意味**: 通信を暗号化して盗聴や改ざんを防ぐ
- **被害例**: HTTPのままだと通信内容（パスワードやトークン）が盗まれる
- **対策**: サイト全体をHTTPSにする

---

## 7. サーバー間通信 vs ブラウザ通信

- **サーバー間通信**:
  - 自社バックエンド → 外部API
  - CORS制約なし
  - 認証情報安全に管理可能
- **ブラウザ通信（フロント直アクセス）**:
  - フロント → 外部API
  - CORS制約あり
  - 認証情報が漏れる危険あり

---

## まとめ

- 業務系システムでは **フロントから外部に直接アクセスしない** のが基本
- CORS, XSS, CSRF などはブラウザや入力を介して発生する攻撃
- 認証情報やデータ加工・監査は **バックエンド経由で管理** することで安全になる
